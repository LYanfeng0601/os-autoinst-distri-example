# openQA 测试脚本编写指南（以 UOS 为例）

本文档说明如何在本仓库（os-autoinst-distri-example）中编写和扩展自动化测试脚本，并以统信 UOS 的引导测试为例贯穿说明。

---

## 1. 概述

### 1.1 运行机制

- **openQA / os-autoinst**：通过 QEMU 等后端启动被测系统，截取屏幕画面，与预设的“针（needle）”进行图像匹配，并根据测试脚本执行按键、输入等操作。
- **本仓库**：是一个“测试发行版”示例，包含测试用例（`tests/*.pm`）、针文件（`needles/`）、入口与调度（`main.pm`、`scenario-definitions.yaml` 等）。

### 1.2 以 UOS 为例的含义

- 测试脚本里会等待/匹配 **UOS 特有界面**（如 UOS 启动菜单），并区分**成功**（如出现启动菜单）与**失败**（如 kernel panic）。
- 编写方式与 openQA 通用写法一致，只是界面和 needle 按 UOS 来设计和命名。

- **新界面/新状态**：在 `needles/` 下新增对应 `.png` + `.json`，在脚本里用 tag 引用。

## 3. 测试脚本基本结构

每个测试模块通常：

1. 继承 `basetest`（来自 os-autoinst）。
2. 使用 `testapi` 与运行环境交互（截图、按键、输入等）。
3. 实现 `run` 子程序，作为该测试的“主流程”。

### 3.1 模板

```perl
# Copyright 2014-2018 SUSE LLC
# SPDX-License-Identifier: GPL-2.0-or-later

use Mojo::Base 'basetest';   # 继承 basetest
use testapi;                 # 使用 testapi

sub run {
    # 1. 等待/断言屏幕
    # 2. 根据匹配结果分支（成功/失败/其他）
    # 3. 发送按键或输入
    # 4. 必要时继续等待下一界面
}

1;
```

- 文件末尾的 `1;` 是 Perl 模块惯例，表示加载成功。

## 4. 常用 testapi 接口

写 UOS 或其它系统的测试时，最常用的是以下几类。

### 4.1 屏幕断言与等待

| 接口 | 用途 |
|------|------|
| `assert_screen $tag, $timeout` | 等待屏幕出现带某 tag 的 needle，超时则失败。 |
| `assert_screen \@tags, $timeout` | 等待出现多个 tag 中的**任意一个**，匹配后可通过 `match_has_tag` 判断是哪一个。 |
| `check_screen $tag, $timeout` | 在指定时间内检测是否出现，不阻塞失败，返回是否匹配。 |
| `match_has_tag($tag)` | 在最近一次 `assert_screen`/`check_screen` 之后，判断是否匹配到该 tag。 |

- **tag**：与 needle 的 `tags` 一致，例如 `'uos-boot-menu'`、`'kernel-panic'`。
- **timeout**：单位秒。例如引导阶段可设 120，进入桌面可设 300。

### 4.2 按键与输入

| 接口 | 用途 |
|------|------|
| `send_key $key` | 发送按键。常用：`'ret'` 回车、`'esc'`、`'tab'`、`'down'`/`'up'` 等。 |
| `send_key_until_needle_match $tag, $key, $timeout, $wait_time` | 反复按某键直到匹配到 needle。 |
| `type_string $string` | 输入字符串（不包含特殊键时使用）。 |

### 4.3 失败与提前结束

| 用法 | 说明 |
|------|------|
| `die "message"` | 立即终止当前测试并标记为失败，常用于检测到严重错误（如 kernel panic）。 |
| `return;` | 正常结束当前测试模块，标记为通过。 |
| `return undef;` | 提前结束、不继续后续断言，可用于“无启动介质”等特殊分支。 |

## 5. Needle（针）的概念与格式

### 5.1 什么是 Needle

- 一个 **needle** = 一张参考截图（`.png`）+ 一个描述文件（`.json`）。
- 测试运行时，当前屏幕会与 needle 进行匹配；匹配到的 **tag** 会在脚本里用于分支判断。

### 5.2 JSON 结构简述（以 UOS 为例）

例如 `needles/uos-boot-menu.json`：

```json
{
  "area": [
    {
      "xpos": 200,
      "ypos": 120,
      "width": 624,
      "height": 380,
      "type": "match",
      "match": 85
    }
  ],
  "tags": ["uos-boot-menu", "bootloader"]
}
```

- **area**：匹配区域（矩形），`xpos`/`ypos` 为左上角，`width`/`height` 为宽高。
- **type**: `"match"` 表示该区域用于图像匹配。
- **match**：匹配阈值（如 85 表示相似度 85%）。
- **tags**：该 needle 对应的标签，脚本里用这些名字在 `assert_screen`、`match_has_tag` 中引用。

通常在实际开发中，needle 会在 openQA 的 needle 编辑器中从真实截图中框选区域、设置 tag 后导出，再放入本仓库的 `needles/` 目录。

## 6. 在 main.pm 中注册测试

新增测试模块后，需要在入口文件中加载，否则不会被调度执行。

**main.pm 示例：**

```perl
use Mojo::Base -strict;
use testapi;
use autotest;

autotest::loadtest 'tests/boot.pm';
# 若有更多测试，继续添加，例如：
# autotest::loadtest 'tests/uos_login.pm';
# autotest::loadtest 'tests/uos_desktop.pm';

1;
```

- `autotest::loadtest 'tests/xxx.pm'` 会按顺序加载并执行对应模块的 `run`。

## 7. 完整示例：UOS 引导测试（boot.pm）解析

下面用仓库中的 `tests/boot.pm` 串起来说明“如何用 testapi + needle 写一段完整流程”。

### 7.1 目标

- 等待三种情况之一出现：**UOS 启动菜单**（成功）、**原版引导界面**（兼容）、**内核恐慌**（失败）。
- 若是 kernel panic，直接失败退出。
- 若是 UOS 启动菜单或原版引导，按回车继续；若已是 UOS 启动菜单则视为本测试通过并结束；否则等待桌面。

### 7.2 对应代码逻辑

```perl
# 等待三种界面之一，最多 120 秒
assert_screen [ 'uos-boot-menu', 'bootloader', 'kernel-panic' ], 120;

# 失败分支：一旦是 kernel panic，立刻 die
if (match_has_tag('kernel-panic')) {
    die 'Kernel panic detected: No working init found. Boot failed.';
}

# 无启动介质时提前结束（原版逻辑）
return undef if match_has_tag('no-boot-media');

# 共同操作：按回车继续启动
send_key 'ret';

# 若当前已是 UOS 启动菜单，本测试目标达成，成功返回
if (match_has_tag('uos-boot-menu')) {
    return;
}

# 否则按原版流程：等待桌面
assert_screen 'desktop', 300;
```

要点小结：

- 用 **数组** 形式的 `assert_screen [ ... ], 120` 实现“多选一”等待。
- 用 **match_has_tag** 区分是哪一个界面，再决定是 `die`、`return undef` 还是 `return`/继续断言。
- 超时时间按阶段设置：引导 120 秒，桌面 300 秒。

### 7.3 当前完整测试步骤（三步骤）

在 `main.pm` 中按顺序加载后，整体流程为：

| 顺序 | 模块 | 作用 |
|------|------|------|
| 1 | `tests/boot.pm` | 等待启动菜单/引导/内核恐慌，按回车继续；若为 UOS 启动菜单则通过，否则等待桌面。 |
| 2 | `tests/uos_login.pm` | 等待登录界面或已进入桌面；若在登录界面则输入用户名/密码（可来自 `USER_LOGIN`、`USER_PASSWORD`）并回车，再等待桌面。 |
| 3 | `tests/uos_desktop.pm` | 等待并断言桌面出现（tag：`uos-desktop` 或 `desktop`），确认系统已进入图形桌面。 |

登录步骤使用的 openQA 变量（在 job 或 schedule 时传入）：

- `USER_LOGIN`：登录用户名，默认 `uos`。
- `USER_PASSWORD`：登录密码，为空则只回车（适用于无密码或自动登录）。

新增的 needle（`uos-login`、`uos-desktop`、`desktop`）需在真实 UOS 下截取对应界面并保存为同名 `.png`，详见 `needles/README.md`。

## 8. 编写新测试的推荐步骤（以 UOS 为例）

### 步骤 1：明确测试目标

例如：“验证 UOS 从启动到出现启动菜单”“验证 UOS 登录后能看到桌面”。

### 步骤 2：确定需要识别的界面

列出所有要“等待/判断”的屏幕状态，并为每个状态起一个 tag 名，例如：

- `uos-boot-menu`
- `uos-login`
- `uos-desktop`
- `kernel-panic`

### 步骤 3：制作或复用 Needle

- 在真实或 openQA 环境中截取对应界面。
- 在 openQA needle 编辑器中框选稳定区域、设置 tag、设置匹配度，导出到 `needles/`（或直接在本仓库中按 5.2 的格式手写 JSON 并配图）。
- 保证 `tags` 与脚本里使用的名字一致。

### 步骤 4：写测试脚本

- 在 `tests/` 下新建 `xxx.pm`，按 3.1 的结构写。
- 用 `assert_screen` / `check_screen` 等待界面，用 `match_has_tag` 分支，用 `send_key` / `type_string` 操作。
- 成功路径用 `return`，失败路径用 `die` 或 `return undef` 视需求而定。

### 步骤 5：在 main.pm 中加载

- 添加一行：`autotest::loadtest 'tests/xxx.pm';`，注意与现有测试的顺序是否符合场景（例如先 boot 再 login 再 desktop）。

### 步骤 6：运行与调试

- **openQA**：在 openQA 中创建 job，使用本仓库作为 CASEDIR，并指定 NEEDLES_DIR 指向本仓库的 `needles`。
- **本地 isotovideo**：使用仓库中 `.github/workflows/isotovideo.yml` 或文档中提到的 isotovideo 方式，在本地或 CI 中跑完整流程。

调试时可根据截图和日志调整 needle 区域、阈值和超时时间。

## 9. 最佳实践与注意事项

1. **超时**：不同阶段给不同 timeout，避免过短导致误报、过长导致挂起。
2. **多分支**：用 `assert_screen \@tags, $timeout` + `match_has_tag` 清晰区分成功/失败/其他分支。
3. **Needle 稳定性**：选择界面中变化少、分辨率变化下仍稳定的区域做匹配；必要时一个界面多个 needle、多个 tag。
4. **注释**：在脚本中为“等待什么”“为什么 die/return”写清注释，便于后续维护（可参考当前 boot.pm 中的注释）。
